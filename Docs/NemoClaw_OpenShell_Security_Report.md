# [기술 분석 보고서] AI 에이전트 보안을 위한 NemoClaw 및 OpenShell 계층 구조 및 보안 메커니즘 비교

본 보고서는 NVIDIA의 AI 에이전트 보안 아키텍처를 구성하는 핵심 컴포넌트인 **NemoClaw**와 **OpenShell**의 구조적 관계, 역할 분담, 그리고 각 레이어별 보안 매커니즘을 상세히 비교·분석하여 안전한 대형언어모델(LLM) 에이전트 런타임 환경을 구축하기 위한 아키텍처 가이드를 제공합니다.

---

## 1. 아키텍처 개요 및 상호 관계

NVIDIA의 AI 에이전트 보안 스택은 에이전트의 자율적 행동이 호스트 시스템이나 내부 네트워크, 그리고 기업의 중요 데이터 자산에 위해를 가하지 못하도록 수직적 방어 계층(Layered Defense)을 형성합니다. 전체 구조는 **[ NemoClaw [ OpenShell [ OpenClaw (에이전트) ] ] ]**의 형태를 취합니다.

* **NemoClaw (거버넌스 및 오케스트레이션 레이어):** 에이전트 외부 환경에서 전체 거버넌스 정책(Blueprint)을 수립하고 데이터 프라이버시를 관리하는 최상위 컨트롤 타워입니다.
* **OpenShell (보안 및 격리 런타임 레이어):** 상위 레이어의 정책을 실시간으로 집행(Enforcement)하며, 커널 수준의 격리와 네트워크 프록시 기술을 통해 물리적·논리적 경계를 세우는 하부 엔진입니다.

두 컴포넌트는 시스템 권한과 데이터를 다루는 레이어가 명확히 분리되어 있어, 하부 에이전트(OpenClaw)가 프롬프트 인젝션(Prompt Injection) 등으로 인해 완전히 탈옥(Escape)하더라도 호스트 시스템과 내부망을 원천적으로 보호할 수 있는 **프로세스 외부 집행(Out-of-Process Enforcement)** 구조를 완성합니다.

## 2. NemoClaw vs OpenShell 핵심 보안 특징 비교

| 비교 항목 | **NemoClaw (최상위 거버넌스 레이어)** | **OpenShell (하부 격리 런타임 레이어)** |
| :--- | :--- | :--- |
| **보안의 초점** | **데이터 프라이버시 및 정책 오케스트레이션** | **호스트 시스템 및 네트워크 인프라 격리** |
| **주요 보호 대상** | 기업의 민감 데이터(PII, 자격 증명), 무결성 | Linux 호스트 커널, 파일 시스템, 내부망 |
| **핵심 보안 기술** | • 민감 정보 탐지 및 마스킹<br>• 프라이버시 라우팅 (Privacy Routing)<br>• Blueprint 무결성 검증 및 CLI 통제 | • 커널 레벨 격리 (Landlock, seccomp)<br>• 네트워크 네임스페이스 (netns)<br>• L7 가로채기 프록시 (OpenShell Gateway) |
| **동작 위치** | 호스트 애플리케이션 프로세스 레벨 | 커널 및 네트워크 샌드박스 내부/경계 레벨 |
| **SSRF 방어 방식** | 입력 URL 텍스트 사전 검증 (`validateEndpointUrl`) | 아웃바운드 트래픽 차단 및 독립 디스패처 핀닝 |
| **자격증명 보호** | Blueprint 정의 및 전달 과정의 자격증명 관리 | 에이전트 프로세스에 API Key 노출 차단 (대리 주입) |

## 3. 컴포넌트별 상세 보안 메커니즘 심층 분석

### 3.1. NemoClaw: 데이터 거버넌스 및 프라이버시 보호
NemoClaw는 에이전트가 처리하는 **'데이터의 내용'**과 **'보안 정책의 선언'**에 집중합니다.

1.  **프라이버시 라우터 (Privacy Router):**
    * 사용자의 입력 프롬프트 및 에이전트 내부 컨텍스트를 실시간으로 스캐닝하여 주민등록번호, 이메일, API 키 등 민감 정보(PII)를 탐지합니다.
    * 콘텐츠의 민감도에 따라 퍼블릭 클라우드 LLM(예: OpenAI)으로의 전송을 차단하고, 사내망에 배치된 로컬 소형 모델(예: Nemotron)로 요청을 동적 라우팅하여 데이터 외부 유출을 원천 방지합니다.
2.  **Blueprint 기반 정책 관리 및 무결성 검증:**
    * 에이전트가 접근 가능한 도구(Tools), 명령어, 네트워크 허용 도메인을 YAML/JSON 형태의 `Blueprint`로 표준화하여 관리합니다.
    * 실행 시점에 Blueprint의 위·변조 여부를 검증하고, 이를 하부 OpenShell 런타임이 이해할 수 있는 로우 레벨 격리 규칙으로 변환 및 하달합니다.
3.  **애플리케이션 레벨 SSRF 1차 방어:**
    * 에이전트가 호출하려는 엔드포인트 URL 문자열을 사전에 분석하여, 프라이빗 IP 대역(RFC 1918)이나 루프백 주소(`127.0.0.1`, `localhost`)가 포함되어 있는지 확인하는 `validateEndpointUrl` 컴포넌트를 운영합니다.

### 3.2. OpenShell: 물리적 격리 및 인프라 방어벽
OpenShell은 에이전트가 동작하는 환경의 **'물리적 경계 규제'**와 **'네트워크 패킷 통제'**를 책임집니다.

1.  **커널 레벨 시스템 샌드박싱 (Sandboxed Environment):**
    * **Landlock (LSM):** Linux 커널의 비특권 접근 제어 메커니즘을 활용하여, 에이전트 프로세스가 지정된 샌드박스 디렉토리(`/sandbox`, `/tmp`) 외의 호스트 파일 시스템에 접근(읽기/쓰기/실행)하는 것을 커널 수준에서 차단합니다.
    * **seccomp:** 에이전트가 취약점을 악용해 권한 상승을 유도하거나 위험한 시스템 콜(Syscall)을 호출하는 행위를 차단합니다.
2.  **L7 프록시 기반 네트워크 격리 (OpenShell Gateway):**
    * 에이전트를 독립된 네트워크 네임스페이스(`netns`)에 격리하여 호스트 및 내부 네트워크망 인터페이스에 직접 접근하지 못하게 만듭니다.
    * 에이전트의 모든 아웃바운드 트래픽은 오직 `OpenShell Gateway`를 통해서만 외부로 나갈 수 있으며, 허용되지 않은 내부망 IP나 차단된 외부 도메인으로의 접근(SSRF)을 네트워크 계층에서 즉각 드롭(Drop)합니다.
3.  **자격 증명 은닉 (Credential Blindness):**
    * 에이전트 자체는 LLM API Key나 외부 서비스 인증 토큰을 전혀 알지 못합니다. 에이전트가 추론 요청을 보낼 때, 중간에 위치한 OpenShell Gateway가 요청을 가로채 안전하게 보관 중이던 API Key를 대리 주입하여 전달합니다. 이로 인해 에이전트가 해킹당하더라도 기업의 상위 자격 증명이 노출되지 않습니다.

## 4. 아키텍처적 결론 및 배포 가이드라인

NemoClaw와 OpenShell은 상호 의존적이면서도 독립적으로 작동할 수 있도록 느슨하게 결합(Loosely Coupled)되어 있습니다.

* **NemoClaw 없이 OpenShell + OpenClaw 조합 운영 가능 여부:** **가능함.**
    * 런타임 격리 및 네트워크 보안(OpenShell)과 에이전트 로직(OpenClaw)만으로도 시스템 안전성은 보장됩니다.
    * 다만, 이 경우 NemoClaw가 제공하던 PII 탐지(Privacy Router), 편리한 통합 CLI(`nemoclaw onboard`), 자동 정책 매핑 혜택을 받지 못하므로 모든 격리 규칙과 게이트웨이 프록시 설정을 수동으로 동기화해야 하는 관리 부담(Operational Overhead)이 발생합니다.

### **최종 요약 권장사항**
엔터프라이즈 환경에서 안전하고 지속 가능한 AI 에이전트를 운영하기 위해서는 **NemoClaw를 활용해 거버넌스 및 자격 증명 관리 효율성을 극대화**하고, 실제 에이전트 구동 환경은 **OpenShell의 강력한 커널/네트워크 격리 기술을 체택하여 이중 방어 체계(Defense in Depth)**를 유지하는 전량적 아키텍처 도입이 필수적입니다.
