# [기술 비교] 주요 AI 에이전트별 샌드박싱 및 격리 메커니즘 분류

본 문서는 시장에 존재하는 다양한 AI 에이전트 및 에이전트 런타임 환경(`Claude Code`, `OpenAI Codex`, `NemoClaw/OpenShell`, `NanoClaw`, `PicoClaw` 등)이 안전한 코드 실행 및 시스템 보호를 위해 채택하고 있는 격리(Sandboxing) 방식을 대분류하여 정리합니다.

---

## 1. 에이전트별 격리 방식 한눈에 보기

AI 에이전트의 격리 메커니즘은 크게 **[1] 인프라 가상화(Docker/VM)** 방식, **[2] OS 커널 네이티브(Landlock/App Sandbox)** 방식, 그리고 **[3] 이들의 장점을 결합한 하이브리드(이중 레이어)** 방식으로 분류됩니다.

| 에이전트 / 런타임 | 핵심 격리 메커니즘 | 메커니즘 분류 | 주요 특징 및 한계 |
| :--- | :--- | :--- | :--- |
| **OpenAI Codex 계열**<br>(Data Analysis 등) | **Docker 컨테이너 / 마이크로VM**<br>(AWS Firecracker 등) | 인프라 가상화 (외벽 중심) | 사용자 코드가 시스템을 완전히 파괴해도 가상 환경만 버리면 되는 구조. 묵직하고 안전함. |
| **Claude Code**<br>(Anthropic CLI) | **OS 네이티브 샌드박싱**<br>(리눅스: Landlock / macOS: App Sandbox) | OS 커널 격리 (내벽 중심) | 개발자 터미널 환경에서 가볍고 빠르게(Docker 없이) 돌면서도 지정된 프로젝트 폴더 외 접근을 차단. |
| **NemoClaw / OpenShell**<br>(OpenClaw, Hermes 등 구동) | **Docker 컨테이너 + Landlock / seccomp**<br>(+ L7 Network Gateway) | **하이브리드 이중 격리**<br>(외벽 + 내벽 융합) | Docker로 가상 방을 만들고, 내부에서 Landlock/seccomp로 수갑을 채운 뒤, L7 프록시로 네트워크(SSRF)까지 삼중 방어. |
| **NanoClaw / PicoClaw** | **클라우드 네이티브 컨테이너**<br>(Kubernetes / Docker 인프라 연동) | 인프라 가상화 (외벽 중심) | 멀티테넌시(여러 유저 분리) 환경이나 마이크로 에이전트 스케일링을 위해 전통적인 컨테이너 오케스트레이션 활용. |
| **ZeroClaw / NullClaw** | **초경량 리눅스 커널 프로세스 격리** | OS 커널 격리 (내벽 중심) | 인프라 오버헤드가 극도로 제한된 임베디드나 에지(Edge) 환경을 타깃으로 하여 Docker 없이 Landlock 위주로 설계. |

## 2. 격리 방식별 상세 분석 및 기술 맵핑

### 🐋 2.1. 인프라 가상화 (Docker / VM) 진영
* **대상 도구:** `OpenAI Codex` 기반 코드 인터프리터 환경, `NanoClaw`, `PicoClaw`
* **작동 원리:** 호스트 운영체제와 완전히 분리된 가상의 미니 컴퓨터 환경(컨테이너 혹은 경량 가상머신)을 매번 독립적으로 새로 생성하여 그 안에서 에이전트를 실행합니다.
* **장점:** 에이전트가 무슨 짓을 하든(예: 악성 패키지 설치, 무한 루프, 포맷 명령 실행) 호스트 OS에는 타격이 0%입니다.
* **단점:** 컨테이너나 VM을 띄우고 빌드하는 데 물리적인 시간과 CPU/메모리 오버헤드가 발생하므로, 가볍고 빠른 로컬 CLI 도구에는 부적합합니다.

### 🔒 2.2. OS 커널 네이티브 격리 진영
* **대상 도구:** `Claude Code` (앤트로픽), 실험적 프로젝트(`ZeroClaw`, `NullClaw` 등)
* **작동 원리:** 무거운 Docker를 띄우지 않고 호스트 OS의 일반 프로세스로 에이전트를 실행하되, 실행 순간 리눅스의 **Landlock**, macOS의 **App Sandbox** 기술을 발동시켜 커널 단에서 팔다리를 묶습니다.
* **장점:** 구동 속도가 0.001초 미만으로 압도적으로 빠르며 컴퓨터 자원을 거의 먹지 않습니다. 개발자 로컬 터미널 환경과 긴밀하게 연동됩니다.
* **단점:** OS 커널 버전에 따라 지원 여부가 갈리며(예: 구형 리눅스나 윈도우 네이티브 환경에서의 제약), 컨테이너만큼 완벽한 멀티테넌시 분리는 어렵습니다.

### 🛡️ 2.3. 하이브리드 이중 격리 진영
* **대상 도구:** `NemoClaw / OpenShell` 인프라 환경
* **작동 원리:** 한 가지 기술만으로는 에이전트의 탈옥이나 SSRF(사내망 해킹) 공격을 완벽히 막을 수 없다는 전제하에 두 진영의 기술을 결합합니다.
    1. **1차 외벽 (Docker):** 인프라 레벨에서 가상 격실을 제공합니다.
    2. **2차 내벽 (Landlock/seccomp):** 컨테이너 내부가 해킹당하더라도 컨테이너 내부 시스템 파일이나 허용되지 않은 디렉토리를 건드리지 못하게 프로세스 권한을 박탈합니다.
    3. **3차 네트워크 방어 (L7 Gateway):** 에이전트가 외부 LLM API와 통신하기 위해 인터넷을 쓰는 순간 발생할 수 있는 사내망 우회 공격(SSRF)을 네트워크 프록시 계층에서 가로채 차단합니다.

## 3. 요약: 아키텍처 선택의 기준

현대 AI 에이전트 생태계는 에이전트의 **사용 목적**에 따라 샌드박싱 아키텍처를 영리하게 선택하고 있습니다.

1. **로컬 개발자 CLI 환경 (`Claude Code` 등):** 속도와 호스트 연동이 중요하므로 **OS 네이티브 커널 격리** 선호.
2. **클라우드 기반 대규모 코드 실행 환경 (`Codex`, `NanoClaw` 등):** 인프라 안정성과 멀티테넌시가 중요하므로 **컨테이너 가상화** 선호.
3. **기업 내 사내망 연동 및 AI 에이전트 런타임 환경 (`OpenShell` 등):** 철저한 보안과 SSRF 방어가 최우선이므로 **Docker 외벽 내부에서 Landlock/seccomp로 한 번 더 잠그는 하이브리드 이중 격리** 선호.
